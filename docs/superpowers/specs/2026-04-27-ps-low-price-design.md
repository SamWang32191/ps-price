# PS 台灣商店歷史低價網站設計

日期：2026-04-27

## 目標

建立一個自用網站，用來同步並查詢台灣 PlayStation Store 的商品與價格歷史。第一版重點是資料可追蹤、同步可排查、歷史低價可信，不做會員、外部通知、跨區比價或公開社群功能。

## 已確認範圍

- 分區：台灣 PlayStation Store。
- 追蹤範圍：全台灣商店可發現商品，包含遊戲、版本包、DLC、追加內容、遊戲內貨幣、訂閱或服務項目。
- 同步頻率：每天一次。
- 使用對象：自己管理用。
- 技術方向：Django + SQLite。
- 爬蟲風險：可接受依賴非官方來源，並把來源結構變動列為已知維護風險。
- 歷史資料範圍：從系統啟用日起累積；PS Store 可見的過去 30 天最低價只作為參考欄位，不混入本系統自己的歷史低價。
- 價格身份：未登入、台灣區、公開可見價格。
- 部署方向：Docker-first，本機先跑，未來可搬到固定主機或雲端。

## 架構

第一版是一個單一 Django 專案，包含管理網站、資料模型、同步命令與 SQLite 資料庫。

核心元件：

- Django server-rendered 管理網站：提供查詢、檢視價格歷史、同步狀態與重跑操作。
- SQLite：保存商品目錄、每日價格快照、同步執行紀錄與錯誤紀錄。
- 同步命令：以 Django management command 實作，由獨立 scheduler container 每天執行。
- PlayStation Store 台灣資料來源 adapter：封裝所有來源抓取與解析邏輯，避免爬蟲細節擴散到頁面或資料模型。

部署路徑：

- 原始碼：`/Users/samwang/code/github.com/SamWang32191/ps-price`
- Docker Compose：`/Users/samwang/dockercompose/ps-price/docker-compose.yml`
- 掛載資料目錄：`/Users/samwang/dockercompose/ps-price/data`

Docker Compose 第一版直接從本機 source path build app image，包含：

- `web` service：Django 網站，只綁定 localhost。
- `scheduler` service：使用同一個 image，負責每日同步。
- 掛載的 `data` 目錄：保存 SQLite、原始回應樣本、失敗原文與備份輸出。

## 資料模型

第一版以「商店商品」為核心單位。一般版、豪華版、DLC、組合包、遊戲內貨幣與訂閱項目各自視為獨立商品，不先嘗試合併成同一款遊戲，避免不同版本或不同商品類型的價格混在同一個歷史低價裡。

主要資料表：

- `StoreProduct`
  - PS Store 商品 ID
  - 商品 URL
  - 標題
  - 平台
  - 來源原始商品類型
  - 本地顯示分類
  - 封面圖
  - 發行商
  - 是否仍可見
  - 連續未在目錄中出現的次數
- `PriceSnapshot`
  - 商品
  - 抓取日期
  - 幣別，第一版固定為 `TWD`
  - 原價
  - 折扣價
  - PS Plus 價格
  - PS Store 顯示的過去 30 天最低價
  - 是否免費
  - 是否不可購買
- `SyncRun`
  - 同步類型
  - 開始與結束時間
  - 狀態
  - 成功筆數
  - 失敗筆數
  - 錯誤摘要
- `SyncError`
  - 所屬同步批次
  - 來源 URL 或商品 ID
  - 錯誤類型
  - 錯誤訊息
  - 是否已重跑成功

本地顯示分類第一版固定為五類：

- 遊戲
- 追加內容
- 組合包
- 訂閱/服務
- 其他

來源原始商品類型必須保留；本地顯示分類可在 Django admin 手動覆寫。

歷史低價從 `PriceSnapshot` 計算，不在第一版手動維護。一般歷史低價只計算可直接購買的 TWD 數字價格；PS Plus 價格、免費狀態、不可購買狀態與 PS Store 過去 30 天最低價分開顯示，不混入一般歷史低價。

## 同步流程

每日同步分成兩段：

1. `sync_catalog`
   - 抓取台灣商店可發現商品清單。
   - 新增尚未存在的商品。
   - 更新標題、平台、商品類型、封面圖等 metadata。
   - 商品消失時先增加「連續未出現次數」，不立即刪除。
   - 記錄每次抓到的商品數、分類數、新增數、消失數，作為完整性監控。

2. `sync_prices`
   - 逐商品抓取當日價格。
   - 寫入或更新同一天同一商品的 `PriceSnapshot`。
   - 單一商品失敗時寫入 `SyncError`，不讓整批同步中斷。
   - 採低併發與固定請求間隔；遇到 rate limit、timeout 或暫時性錯誤時退避重試。

同步需要具備 idempotency：同一天重跑不產生重複價格快照。

全商店完整性採「可觀測最佳努力」：系統要記錄覆蓋數量與異常變化，逐步改善可發現商品覆蓋率，但 MVP 不宣稱已達絕對 100% 完整。

原始爬蟲回應不全量永久保存。價格快照永久保存；原始回應只保存失敗案例與少量成功樣本，用於排查來源格式變動。

## 錯誤處理

- 來源暫時失敗時記錄錯誤並繼續處理其他商品。
- 商品短暫消失時只標記為可能下架，連續 7 次每日目錄同步都未出現後才視為下架。
- 每次同步都留下 `SyncRun`，讓 Dashboard 能判斷資料是否新鮮。
- 爬蟲來源格式改變時，錯誤應集中出現在 adapter 與同步紀錄，方便定位。
- 同步失敗只在 Dashboard 與同步紀錄中顯示；第一版不做 email、LINE、Telegram 或桌面通知。

## 測試策略

第一版至少測試：

- 價格解析：原價、折扣價、免費、PS Plus 價。
- 歷史低價計算：一般價格與 PS Plus 價格分開。
- 同步 idempotency：同一天重跑不產生重複快照。
- 來源失敗：單一商品失敗不讓整批同步中斷。
- 商品狀態：商品消失時先標記可能下架，不刪除歷史價格。
- 本地分類覆寫：來源分類不變，本地顯示分類可被 admin 覆寫。

第一個實作里程碑是 crawler spike。完成標準：

- 能抓到台灣商店目錄清單。
- 能解析一批樣本商品。
- 樣本至少涵蓋原價、折扣價、免費、PS Plus 價與不可購買案例。
- 留下來源 fixture，讓價格解析測試可重跑。

## 非目標

第一版不做：

- 一般使用者會員系統或公開登入系統。
- 願望清單。
- 降價通知。
- 跨區比價。
- 公開首頁或 SEO。
- 自動合併同款遊戲的不同版本。
- 多年既有歷史回填。
- 全量原始爬蟲回應永久保存。

## 成功標準

- 每天可同步台灣商店商品目錄與價格。
- 查詢單一商品時，可以看到目前價格與歷史低價。
- 查詢單一商品時，可以看到本系統價格資料起始日。
- 商品可以依五類本地顯示分類分頁或篩選。
- 同步失敗時，可以在網站上看到失敗原因並重跑。
- 同一天重跑同步不會製造重複價格資料。
- PS Plus 專屬價格不會混入一般歷史低價。
- Docker Compose 可在本機從 source path build 並使用指定 data 目錄保存資料。
